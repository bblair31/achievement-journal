## Summary

Production incident on 2026-03-13: workup-service became unresponsive due to PostgreSQL connection pool exhaustion (8 instances × 25 pool = 200, saturating `max_connections=200` on 1 vCPU Cloud SQL).

**This PR fixes the pool config and addresses pre-existing issues found during investigation:**

- **Pool config**: `max: 25→15`, `idleTimeoutMillis: 15s→10s`, `connectionTimeoutMillis: 10s→5s`
- **Broken transactions**: `query()` acquired/released a fresh client per call, so `BEGIN`/`INSERT`/`COMMIT` ran on different connections — transactions were no-ops. Rewritten with `AsyncLocalStorage` so `query()` auto-detects transaction context. 10 call sites converted to `withTransaction()`.
- **SQL injection**: 11 call sites used string interpolation (`WHERE id = '${id}'`) instead of parameterized queries (`WHERE id = $1`). All fixed.
- **`connect()` race condition**: Concurrent cold-start calls could create duplicate pools. Fixed with promise singleton pattern.
- **Performance**: `getMostRecentApplication()` with `LIMIT 1` for bulk reroute (was fetching all applications). `getTopMatchHighDemandOpportunities` moved into `Promise.all` (was blocking sequentially).
- **Error handling**: ROLLBACK failure now preserves original error and destroys poisoned client. Pool error handler uses `logger.error` with structured fields.

#### #161: Reduce effects of old unrelated apps vs newer apps' roles - ✅ merged ([link](https://github.com/WorkUp-Health/WorkUp-BigQuery/pull/161))

- **Created:** Mar 20, 2026

**Description:**
[CU-86afzdz2v](https://app.clickup.com/t/86afzdz2v)

When we infer what role a candidate is interested in from their past applications, we treat every application equally. A single receptionist application from 2 years ago counts the same as 5 recent RN applications. Recruiters see candidates matched to roles they've clearly moved on from.

What we're changing: If we can clearly identify a candidate's primary role AND they only applied to a secondary role once or twice over a year ago, we stop matching them to the secondary role.

**It took longer than I'd like to come to this solution and I tried many others like timeframe windows, decay, role hierarchies/career progressions, etc but this approach was the simplest and achieved the best and least risky results. Don't want to overcorrect when we know we want to improve role preference tagging anyway and hopefully fewer candidates fall into this "application data only" tier 4 in the future**
